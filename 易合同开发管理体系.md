## 第二部分：重构方案

### 2.1 重构原则

```
1. 不中断业务：重构过程中保持主分支可部署
2. 增量改动：每次只改一个模块，改完测试再继续
3. 按影响范围排序：从底层到上层
   数据模型 → API → 组件 → 页面
4. 去重优先：先清理重复文件，减少心智负担
```

### 2.2 分阶段重构路径

## 第三部分：开发制度

### 3.1 分支管理

```
main           ← 生产环境，只从 release 分支合并
  └── dev      ← 日常开发，所有 feature 分支合并到此
       ├── feature/data-model     ← 数据模型改造
       ├── feature/search         ← 搜索功能
       ├── feature/import         ← Excel导入
       ├── feature/onboarding     ← 用户引导
       └── fix/*                  ← 修复分支
```

### 3.2 提交规范

```
格式：<类型>: <简短描述>

类型：
  feat     → 新功能
  fix      → Bug修复
  refactor → 重构
  style    → 代码格式（不影响功能）
  docs     → 文档
  test     → 测试
  chore    → 杂项（清理文件/配置修改）

示例：
  feat: 合同新增收支方向标记
  fix: 修复AI解析超时未正确提示
  refactor: 抽取统一API错误处理
  chore: 删除_duplicates_raw目录
```

### 3.3 代码审查清单（每次提交前自检）

```
□ 无 TypeScript 错误（npm run type-check）
□ 无 ESLint 警告（npm run lint）
□ 新增 API 有错误处理
□ 数据库 migration 已生成
□ 无重复文件
□ 控制台无 console.log（用 logger 替代）
□ 敏感信息（密钥/密码）不在代码中
```

### 3.4 开发环境规范

```
本地开发：
  ─ 端口: 3001（已有配置）
  ─ 数据库: SQLite（开发用）
  ─ AI模型: 本地 Ollama（可选）或 OpenAI API

命令：
  npm run dev        → 开发服务器
  npm run build      → 构建检查（含 TS + ESLint）
  npm run lint       → ESLint 检查
  npm run db:push    → 数据库 Schema 同步
  npm run db:migrate → 生成迁移
  npm run type-check → TypeScript 检查（需要加到 package.json）
```
